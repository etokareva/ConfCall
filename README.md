# ConfCall Scheduler

ConfCall Scheduler — приложение для планирования видеовстреч программных комитетов конференций. Участники указывают доступность, система ищет общий свободный слот, создаёт встречу и сохраняет ссылку на видеозвонок.

**Теги для навигации:** `#angular` `#nestjs` `#mysql` `#groups` `#availability-calendar` `#booking` `#meetings` `#invite-auth` `#profile` `#i18n` `#telemost` `#cdk-overlay` `#design-tokens`

## Возможности

- Настройка личной доступности в month-view календаре: разовые и повторяющиеся слоты создаются через диалог выбранного дня.
- Проверка пересечений при создании слотов: разовые слоты объединяются с разовыми, регулярные — с регулярными, а конфликт разового и регулярного слота требует выбора области действия.
- Отображение запланированных встреч в календаре доступности с модальным просмотром деталей; прошедшие встречи визуально приглушаются и зачеркиваются.
- Группы пользователей: создание групп, приглашения по email и добавление участников после подтверждения.
- Автоматический поиск пересечения доступности участников выбранной группы при изменении даты, диапазона дат или длительности.
- Создание встреч с участниками и ссылкой Яндекс Телемоста.
- Страница `Мои встречи` с фильтрами: будущие, новые, прошедшие, отменённые, все.
- Бейдж новых встреч в навигации и подсветка новых встреч при первом открытии.
- Публичные ссылки бронирования создаются на `/book` для выбранной группы и выбранной части участников, а отображаются в `/booking-links` только у создателя. В таблице созданных ссылок доступны только копирование URL и удаление.
- После публичного бронирования встреча появляется у всех участников группы, гость получает письмо со ссылкой на видеозвонок и отдельной ссылкой отмены.
- Если встреча отменена организатором или гостем по ссылке, в карточке встречи показывается, кто её отменил.
- Публичная ссылка может быть выключена и снова включена. При переходе по выключенной ссылке гость видит понятное сообщение и кнопку повторной проверки; после повторного включения та же slug-ссылка снова доступна для просмотра и бронирования.
- Регистрация только по ссылке приглашения в группу: если пользователь ещё не зарегистрирован, он создаёт учётную запись прямо по invite-ссылке и сразу попадает в нужную группу.
- Регистрация в два шага: сначала создаётся pending-аккаунт, затем на email уходит ссылка подтверждения, и только после перехода по ней учётная запись активируется. Если email ещё не подтверждён, повторная регистрация с тем же адресом переотправляет ссылку, а не создаёт дубликат.
- Вход по email и паролю, повторная отправка письма подтверждения и сброс пароля через email-ссылку.
- Профиль пользователя: отображаемое имя, фото и язык интерфейса. Если ссылка на аватар доступна, интерфейс показывает изображение вместо инициалов.
- Runtime-локализация интерфейса на русский, английский, испанский и китайский с сохранением языка в профиле.
- Backend ошибки и бизнес-сообщения API приходят как ключи локализации, а фронтенд переводит их по текущему языку.
- Dev-auth для локальной разработки.
- Playwright screenshots через Docker для визуальной проверки интерфейса.
- Playwright + axe-core для автоматического accessibility audit WCAG A/AA на `/login` и основных авторизованных маршрутах.
- Единая система дизайн-токенов для цветов, семантических маркеров календаря, отступов, радиусов, размеров контролов и типографики.
- Стилизованный выбор даты и длительности на странице бронирования.

Подробнее по инструментам: [TECH_STACK.md](./TECH_STACK.md)

Для работы с AI-агентами: [AI_ONBOARDING.md](./AI_ONBOARDING.md) описывает, как быстро познакомить агента с проектом, какие документы давать перед задачей и какие проверки запускать.

Модальные окна используют общий каркас `ccs-modal-shell`. Простые подтверждения и сообщения вызываются через `ModalService`; содержательные диалоги переиспользуют shell для единого header, close-кнопки, footer, focus/z-index поведения и визуального стиля.

При переходе на другой маршрут приложение очищает transient UI: закрывает все CDK Dialog модалки и удаляет активные/ожидающие toast-уведомления. Это поведение находится в `frontend/src/app/app.component.ts`; новые модалки должны открываться через CDK Dialog/`ModalService`, а toast-уведомления — через `ToastService`, чтобы автоматически закрываться при навигации.

Маршрутизация и доступ:

- `/` — публичная главная страница с кратким описанием продукта.
- `/login` и `/book/:slug` доступны без авторизации.
- Все остальные страницы требуют входа и редиректят на `/login`, если пользователь не залогинен.
- Авторизованный пользователь, открыв `/login`, возвращается в `returnUrl` или на `/dashboard`.

Реестр UI-компонентов и правила переиспользования: [COMPONENTS.md](./COMPONENTS.md).

## Архитектура

```text
frontend/              Angular SPA
server/                NestJS API
db/                    Drizzle schema and relations
docker-compose.yml     MySQL and Playwright services
e2e/                   Playwright specs and screenshot checks
artifacts/screenshots/ Generated UI screenshots
```

Основные backend-модули:

- `auth` — dev-login, текущий пользователь, обновление профиля, регистрация по invite-ссылке.
- `group` — группы, аватар группы, участники групп, приглашения в группы.
- `availability` — хранение доступности и поиск пересечения свободного времени.
- `meeting` — создание, список, отмена, отметка новых встреч просмотренными.
- `booking` — публичные ссылки бронирования и создание встречи гостем.
- `video` — создание конференции через Яндекс Телемост API.
- `db` — Drizzle provider для MySQL.

Основные frontend-разделы:

- `/login` — вход разработчика, вход по email и паролю, повторная отправка письма подтверждения и запрос сброса пароля.
- `/` — публичная страница с кратким описанием приложения и кнопками входа или регистрации.
- `/dashboard` — UI-раздел `Главная`, страница `Обзор встреч`: ближайшие встречи, новые события и быстрые действия.
- `/availability` — month-view календарь доступности: orchestration-страница, которая собирает `ccs-availability-month-grid`, `ccs-availability-day-details` и `ccs-availability-event-dialog`.
- `/groups` — управление группами, аватаром группы, приглашениями и участниками. Создание группы, редактирование основной информации и отправка приглашений открываются компактными действиями в заголовках и модальных окнах.
- `/book` — трёхшаговый сценарий выбора группы, участников и доступного времени. На шаге `Доступное время` reactive form автоматически пересчитывает пересечения по одной дате или диапазону дат при изменении даты, режима или длительности; отдельной кнопки `Показать варианты` нет. Для диапазона отправляется один API-запрос с `startDate`/`endDate`, а не серия запросов по дням. Выпадающий календарь подсвечивает даты со свободным временем: для одного участника — его свободные дни, для нескольких — даты с общими пересечениями. Если пересечения найдены, становится доступна кнопка создания публичной ссылки, которая открывает модальное окно с предзаполненной длительностью.
- `/booking-links` — список публичных ссылок, созданных текущим пользователем. Последняя колонка намеренно лаконичная: только копировать URL и удалить ссылку.
- `/meetings` — список встреч с фильтрами и подсветкой новых.
- `/profile` — профиль пользователя, отображаемое имя, фото и язык приложения.
- `/access` — устаревший раздел; сценарии публичного бронирования перенесены в `/book`.
- `/settings` — совместимый редирект на `/profile`.
- `/book/:slug` — публичная страница бронирования по slug-ссылке. Для групповой ссылки показывает аватар группы, участников, подсветку дат со свободным временем, слоты и форму гостя.
- `/book/cancel/:token` — публичная страница отмены встречи гостем по ссылке из письма.

Глобальные UI-токены находятся в [frontend/src/styles/\_tokens.scss](./frontend/src/styles/_tokens.scss). Компонентные SCSS-файлы должны использовать CSS variables из этого файла вместо локальных hardcoded-цветов и повторяющихся размеров. Для календаря доступности главный визуальный контраст строится между свободным и занятым временем через `--color-slot-free`, `--color-slot-busy` и соответствующие overlay-токены. Разница между разовым и повторяющимся свободным слотом не должна получать отдельный сильный акцент.

Angular-селекторы компонентов и UI-директив используют префикс `ccs`, например `<ccs-navbar>` и `ccsTooltip`.

## База данных

Таблицы описаны в [db/schema.ts](./db/schema.ts):

- `users`
- `availability_slots`
- `groups`
- `group_members`
- `meetings`
- `meeting_participants`
- `booking_links`
- `booking_link_participants`
- `group_invitations`

`users.locale` хранит выбранный язык интерфейса: `ru`, `en`, `es` или `zh`.

`groups.avatar` хранит URL изображения группы. На публичной странице бронирования изображение показывается в header; если URL пустой или картинка не загрузилась, используется иконка `users`.

`booking_links.userId` хранит создателя ссылки; `GET /api/booking/links` возвращает только ссылки текущего пользователя, а удаление доступно только создателю. `booking_links.groupId` связывает публичную ссылку с группой. Ссылки создаются в `/book` после выбора группы, участников и доступного времени, а отображаются в `/booking-links`; `isActive=false` временно запрещает публичное бронирование и возвращает API-ключ `booking.link_inactive`.

API дат использует единый контракт date-only ISO `YYYY-MM-DD`. UI может показывать локализованный формат, но frontend-сервисы должны отправлять в query только `date`, `startDate` и `endDate` в формате `YYYY-MM-DD`.

`booking_link_participants` хранит выбранных участников публичной ссылки. Это позволяет создать ссылку не на всю группу, а только на нужную часть участников. Для старых ссылок без записей участников backend использует fallback на всех участников группы.

`meeting_participants.cancelToken` хранит одноразовую публичную ссылку отмены для гостя, забронировавшего встречу по slug-ссылке. `meetings.cancelledByName`, `meetings.cancelledByEmail` и `meetings.cancelledAt` фиксируют, кто отменил встречу.

Для локальной разработки используется MySQL из Docker Compose:

```text
host: localhost
port: 3306
database: confcall_scheduler
user: user
password: password
```

## Переменные окружения

Минимальный `.env`:

```env
MYSQL_DATABASE=confcall_scheduler
MYSQL_USER=app_user
MYSQL_PASSWORD=change_me_mysql_password
MYSQL_ROOT_PASSWORD=change_me_mysql_root_password
DATABASE_URL=mysql://app_user:change_me_mysql_password@localhost:3306/confcall_scheduler
```

Для реального создания видеовстреч:

```env
TELEMOST_OAUTH_TOKEN=oauth_access_token
```

Токен должен быть выпущен пользователем организации Яндекс 360 для бизнеса и иметь scope:

```text
telemost-api:conferences.create
```

Если токен отсутствует или Яндекс его отклоняет, backend возвращает человекопонятную ошибку настройки.

## Авторизация в разработке

В dev-режиме доступна кнопка `Dev-вход администратора` на `/login`.

После входа пользователь может открыть `Настройки`, создать группу и отправить приглашения на список email. Новый пользователь переходит по ссылке из письма `/invite?token=...`: если аккаунта ещё нет, он регистрируется прямо по invite-ссылке; если аккаунт уже есть, он может войти и принять приглашение. После принятия приглашения пользователь автоматически добавляется в группу. Если письмо не пришло, его можно отправить повторно через интерфейс группы. Вход по email и паролю остаётся доступен после подтверждения адреса и установки пароля. Если пароль забыт, на `/login` доступен запрос нового письма для сброса пароля и страница `/reset-password`.

Для реальной отправки писем приложение использует SMTP Яндекс Почты для домена. Нужны `SMTP_HOST=smtp.yandex.ru`, `SMTP_PORT=465`, `SMTP_SECURE=true`, `SMTP_USER`, `SMTP_PASS`, `MAIL_FROM` и `APP_PUBLIC_URL`. Для локальной настройки используйте существующий ящик вида `mailer@example.com` или alias внутри вашего домена. Если доменная почта или DNS-записи `MX/SPF/DKIM` настроены неверно, письмо может не дойти или попасть в спам.

Подробный сценарий: [AUTH_FLOW.md](./AUTH_FLOW.md)
