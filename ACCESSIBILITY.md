# Accessibility Guidelines

Этот документ фиксирует правила доступности для ConfCall Scheduler. Он предназначен и для разработчиков, и для AI-агентов, которые меняют интерфейс.

Для быстрого входа в проект вместе с этим документом читать [AI_ONBOARDING.md](./AI_ONBOARDING.md) и [COMPONENTS.md](./COMPONENTS.md). Если меняется сложный виджет, модалка, dropdown, календарь, stepper или форма, доступность проверяется как часть задачи, а не отдельным последующим этапом.

## Цель

Базовый стандарт проекта — WCAG 2.2 уровня AA. Приоритеты: клавиатурная навигация, понятные имена контролов, видимый фокус, читаемый контраст, корректные состояния форм, предсказуемые модальные окна, тосты и dropdown.

## Общие правила

- Каждый интерактивный элемент должен быть доступен с клавиатуры и иметь понятное accessible name.
- Keyboard navigation: все интерактивные элементы должны быть достижимы через `Tab`, `Shift+Tab` и, для сложных виджетов, через стрелки. Это проверяется частично автоматически и обязательно вручную на ключевых сценариях.
- Keyboard traps: в приложении не должно быть элементов, которые удерживают фокус и не дают уйти дальше. Исключение — открытое модальное окно, где фокус удерживается внутри до закрытия, после чего возвращается к инициатору.
- Focus indicator: каждый интерактивный элемент должен иметь видимый индикатор фокуса. Это проверяется вручную, потому что автоматический аудит не видит все визуальные состояния.
- Tab order: порядок `Tab` должен совпадать с визуальной и смысловой логикой интерфейса. Не использовать положительные `tabindex`; для сложных виджетов применять roving tabindex.
- Input focus: фокус не должен неожиданно перемещаться без действия пользователя. Допустимый перенос фокуса — по явному открытию модалки, dropdown, календаря или после смены маршрута на основной landmark.
- Если элемент выглядит как кнопка и выполняет действие, использовать `<button type="button">`. Если элемент ведёт на маршрут или URL, использовать ссылку.
- Не полагаться только на цвет: статус должен иметь текст, aria-label, tooltip или другой нецветовой признак.
- Видимый `:focus-visible` обязателен для ссылок, кнопок, полей, custom dropdown и календарных действий.
- Минимальная практичная зона нажатия — 24×24 CSS px, для основных действий лучше 44×44.
- Иконки, которые только украшают текст, должны иметь `aria-hidden="true"`. Иконки без видимого текста должны получать `aria-label` на кнопке или ссылке.
- Тексты для `aria-label`, тултипов, тостов, ошибок и кнопок должны проходить через i18n-ключи.

## Структура страницы

- Основное содержимое приложения находится внутри `main#main-content`.
- В приложении должен быть только один основной landmark `main`; feature-страницы не должны вкладывать собственный `<main>` внутрь router-outlet.
- После смены маршрута фокус переводится на `main#main-content`, чтобы screen reader не оставался на старом пункте меню.
- В навигации должен быть skip link `Перейти к основному содержимому`.
- Верхнее меню должно быть `<nav>` с локализованным `aria-label`.
- Активный пункт навигации должен объявляться через `aria-current="page"`.
- На странице должен быть один основной `h1`; секции ниже используют последовательные заголовки.
- Не использовать пустые заголовки или заголовки только ради размера текста.

## Формы

- У каждого поля должен быть видимый `<label>` или явно связанный accessible name.
- Видимый label связывать с полем через `for/id`, а не только располагать рядом.
- Для email, имени и одноразовых кодов использовать подходящие `autocomplete` значения.
- Ошибка должна быть текстом рядом с полем и связываться через `aria-describedby`, если ошибка относится к конкретному полю.
- Для обязательных полей использовать визуальное пояснение и, где уместно, `required` / `aria-required`.
- При серверной ошибке показывать человекопонятное сообщение, не только код.
- При успешном сохранении использовать toast или короткий статус, а не менять контекст без объяснения.
- Не запускать навигацию или сохранение только от изменения значения поля. Значимые изменения выполняются по явному действию пользователя.

## Dropdown и меню

- Trigger dropdown должен иметь `aria-haspopup="menu"`, `aria-expanded` и `aria-controls`.
- Контейнер меню получает `role="menu"`, элементы внутри — `role="menuitem"`, если это меню действий.
- Меню закрывается по `Escape`, клику вне меню и потере фокуса.
- Tooltip на trigger не должен оставаться видимым при переходе в submenu.
- Если выбор работает как единственный активный вариант, использовать `radiogroup`/`radio` или нативный radio-control. Если выбор множественный, оставлять настоящие кнопки с `aria-pressed`.

## Диалоги

- Модальные окна открывать через общий сервис и CDK Dialog.
- Любая новая модалка должна быть CDK Dialog. Не создавать локальные fixed-backdrop или условно отрисованные `@if`-модалки внутри feature-страницы.
- Все содержательные диалоги должны использовать `ccs-modal-shell`, чтобы заголовок, close-кнопка, header actions и footer были одинаковыми.
- `ccs-modal-shell` задаёт `role="dialog"`, `aria-modal="true"`, `aria-labelledby` и focus trap. Если у диалога есть описание, передавать его через `describedBy`, чтобы сформировать `aria-describedby`.
- При открытии модального окна фокус должен переходить внутрь окна: на первый осмысленный контрол или на сам dialog, если немедленный фокус на поле мог бы неожиданно изменить контекст.
- Пока модальное окно открыто, `Tab` и `Shift+Tab` остаются внутри него; `Escape` закрывает окно там, где это безопасно.
- После закрытия модального окна фокус должен возвращаться на элемент, который его открыл, если он ещё существует.
- В диалоге должен быть понятный заголовок и явные действия.
- Нативные `alert` и `confirm` не использовать.
- Закрытие по `Escape` допустимо, если это не приводит к потере несохранённых данных без предупреждения.
- При переходе на другой маршрут открытые модальные окна должны закрываться, чтобы focus trap и скрытый контент старой страницы не оставались активными после навигации.

## Toast и status messages

- Обычные уведомления используют `role="status"` и polite live region.
- Ошибки используют `role="alert"` или assertive-семантику только когда действие пользователя действительно требует внимания.
- Toast должен иметь доступную кнопку закрытия с локализованным `aria-label`.
- Не показывать важную информацию только в toast, если она нужна для продолжения сценария. Дублировать её в контексте страницы или диалога.
- При навигации toast-уведомления очищаются: после перехода screen reader не должен объявлять устаревшие сообщения со старого экрана.

## Tooltip

- Tooltip — вспомогательная подсказка, а не единственный способ понять действие.
- Для icon-only действий tooltip дополняет `aria-label`, но не заменяет его.
- Tooltip не должен забирать фокус: фокус остаётся на кнопке, ссылке или поле, которое вызвало подсказку.
- Tooltip должен скрываться по `Escape`, `mouseleave` и `focusout`.
- Текст tooltip должен быть коротким и локализованным.

## Календарь доступности

- Действия в ячейке дня должны быть кнопками с понятными `aria-label`.
- Маркеры занятых встреч не должны выглядеть как кнопки, если они не кликабельны.
- Цветовые маркеры свободного и занятого времени должны иметь текстовое объяснение в легенде.
- Календарные поля ввода должны объявлять открытие popup через `aria-haspopup` и отражать состояние через `aria-expanded`, если виджет это поддерживает.
- При клике по дню открывается просмотр расписания дня; добавление слота выполняется отдельной кнопкой.
- Встречи, которые блокируют время, должны отображаться как занятое время и не предлагаться для бронирования.

## Проверка перед завершением задачи

- Пройти основную страницу с клавиатуры: `Tab`, `Shift+Tab`, `Enter`, `Space`, `Escape`.
- Проверить, что focus ring виден и не перекрыт sticky/header/overlay элементами.
- Проверить icon-only кнопки: у каждой есть `aria-label` и tooltip.
- Проверить формы: поля имеют labels, ошибки понятны, submit не теряет данные.
- Проверить live-сообщения: toast и ошибки объявляются screen reader-семантикой.
- Запустить автоматический axe-аудит: `npm run a11y`.
- Если локальный Chromium не стартует в текущей среде, запускать audit через `docker compose run --rm playwright npm run a11y`.
- Запустить `npm run format` и `npm run check`.

## Ручная проверка через Accessibility Insights for Web

Автоматический `axe` в Playwright ловит часть проблем, но не заменяет ручной проход по пользовательскому сценарию. Для страниц со сложными виджетами дополнительно использовать Chrome extension **Accessibility Insights for Web**.

Минимальный ручной сценарий:

- Открыть страницу в Chrome и запустить **FastPass**.
- Проверить найденные WCAG-ошибки: contrast, `aria-allowed-attr`, missing labels, duplicate ids, invalid roles.
- Для каждого нарушения смотреть target, snippet и suggested fix; исправлять причину в общем компоненте, если проблема повторяется в нескольких местах.
- После исправления повторно запустить FastPass на той же странице.
- Для сложного сценария включить ручную проверку **Tab stops** и пройти интерфейс клавиатурой.
- Проверить, что focus order совпадает с визуальной логикой страницы.
- Проверить отсутствие keyboard traps: из dropdown, календаря, меню и модального окна можно выйти ожидаемым способом; внутри модального окна фокус не уходит на страницу за ним.
- Проверить input focus: фокус не прыгает при загрузке данных, пересчёте результатов, ошибках формы и переключении режимов без явного действия пользователя.
- Проверить, что календарь, dropdown, меню, модальные окна и списки доступны через клавиатуру.
- Проверить, что пользователь может выполнить ключевой сценарий без мыши: открыть поле, выбрать значение, отправить форму, закрыть диалог.

Что проверять вручную особенно внимательно:

- Контраст текста ошибок, muted text, disabled states, подсказок и мелких подписей.
- ARIA-атрибуты на кастомных контролах: атрибут должен поддерживаться текущей ролью элемента.
- `input` с popup-календарём: ручной ввод даты должен работать с клавиатуры, а открытый календарь должен поддерживать стрелки, `Enter`/`Space` и `Escape`.
- Icon-only actions: у кнопки должен быть `aria-label`, а tooltip не должен быть единственным источником смысла.
- Модальные окна: начальный фокус, закрытие по `Escape`, возврат фокуса после закрытия, отсутствие фокуса за модалкой.
- Toast и ошибки: важные сообщения должны быть объявляемыми через `role="status"` или `role="alert"` и продублированы в контексте, если без них нельзя продолжить.

## Что уже внедрено

- `main#main-content` вокруг Angular router outlet.
- Skip link к основному содержимому.
- Перенос фокуса на `main#main-content` после смены маршрута.
- Локализованный `aria-label` для основной навигации.
- Активные пункты верхнего меню помечаются через `aria-current="page"`.
- Screen-reader текст с количеством новых встреч рядом с визуальным бейджем.
- Icon-only пункты меню имеют `aria-label`, скрытый текст и tooltip.
- Dropdown аккаунта имеет `aria-haspopup`, `aria-expanded`, `aria-controls`, `role="menu"` и закрытие по click outside, `Escape` и потере фокуса.
- Общая модалка имеет dialog-семантику и связь заголовка/описания.
- Все модальные окна открываются через CDK Dialog; `ccs-modal-shell` централизованно задаёт dialog-семантику, focus trap и initial focus для содержательных модальных окон.
- Toast использует `role="status"` для обычных сообщений и `role="alert"` для ошибок.
- Формы входа используют `tablist`/`tabpanel` семантику для переключения сценариев входа, регистрации и сброса пароля.
- Формы входа и ключевые поля бронирования имеют связанные `label for/id`.
- Страницы публичного бронирования и списка ссылок не создают вложенные `main` landmarks и объявляют загрузку/ошибки через `role="status"` и `role="alert"`.
- Popup календаря в `ccs-date-input` объявляется через `aria-haspopup="dialog"`; состояние `aria-expanded` находится на кнопке открытия календаря, а не на текстовом input.
- Выпадающий календарь `ccs-date-input` поддерживает ручной ввод даты и keyboard-сценарий выбора дня: `ArrowDown` открывает календарь, стрелки перемещают фокус по дням, `Home`/`End` переходят к краям недели, `PageUp`/`PageDown` двигают фокус по месяцам, `Enter`/`Space` выбирают дату, `Escape` закрывает календарь.
- Контраст calendar day numbers и subtitle в selectable-card-grid проверен и доведён до прохождения axe WCAG A/AA.
- Контраст ошибок на публичной странице бронирования усилен до WCAG AA: мелкий текст ошибок использует более светлый danger-token.
- Выбор группы в бронировании размечен как `radiogroup`, выбор участников — как набор toggle-кнопок с `aria-pressed`.
- Playwright содержит axe-аудит WCAG A/AA для `/login` и основных авторизованных разделов; актуальная проверка проходит через Docker Playwright-контейнер.
