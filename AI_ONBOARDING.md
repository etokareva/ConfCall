# AI Onboarding — ConfCall Scheduler

Теги: `#ai-onboarding` `#agents` `#architecture` `#frontend` `#backend` `#ux` `#docs`

Этот документ помогает быстро и безопасно познакомить AI-агента с проектом перед задачей. Его стоит давать агенту вместе с `AGENTS.md`, если нужно не просто точечно исправить баг, а продолжать развитие продукта.

## Что Сообщить AI-Агенту В Первом Сообщении

Минимальный контекст:

```text
Работай в /home/kate/confcall-scheduler.
Сначала прочитай AGENTS.md и AI_ONBOARDING.md.
Проект: Angular 19 SPA + NestJS + MySQL, приложение для планирования видеовстреч.
Соблюдай существующие правила: ccs-префикс, runtime i18n, design tokens, reactive forms, CDK overlay/dialog, доступность.
После изменений запускай npm run format и npm run check.
Не откатывай чужие изменения.
```

Если задача про UI/UX, добавить:

```text
Перед правками проверь COMPONENTS.md, ACCESSIBILITY.md и существующие паттерны в соседних страницах.
Не добавляй новые визуальные акценты без необходимости.
Сначала убирай дублирование и лишние блоки, потом стилизуй.
```

Если задача про архитектуру или доменную логику, добавить:

```text
Перед изменениями проверь ADR.md, db/schema.ts и соответствующий backend/frontend сервис.
Если меняется контракт API, обнови общие типы и документацию.
```

## Быстрый Порядок Знакомства

1. Прочитать [AGENTS.md](./AGENTS.md): обязательные правила, маршруты по тегам, стиль разработки.
2. Прочитать этот файл: как входить в проект и какие документы подключать по типу задачи.
3. Открыть [README.md](./README.md): продукт, возможности, основные маршруты, запуск.
4. Открыть [COMPONENTS.md](./COMPONENTS.md), если задача касается UI.
5. Открыть [ACCESSIBILITY.md](./ACCESSIBILITY.md), если меняются интерактивные элементы, модалки, меню, календарь, формы или тосты.
6. Открыть [ADR.md](./ADR.md), если меняются архитектура, API-контракты, база, интеграции, модалки, i18n или deployment.
7. Открыть [TECH_STACK.md](./TECH_STACK.md), если нужно понять, зачем используется библиотека или инструмент.
8. Смотреть код только после документов: сначала ближайший feature-модуль, затем shared/core сервисы.

## Правила Быстрого Анализа Кода

- Начинать поиск с `rg`, не с ручного просмотра всего дерева.
- Для feature-задачи сначала открыть `.ts`, `.html`, `.scss` одной страницы или компонента.
- Проверить, нет ли уже shared-компонента или сервиса для нужного паттерна.
- Не добавлять новый паттерн, если задача закрывается `ccs-modal-shell`, `ccs-date-input`, `ccs-time-select`, `ccs-dropdown`, `ccsTooltip`, `ToastService`, `ModalService`.
- Для UI сначала определить главный пользовательский сценарий и убрать дублирующую информацию.
- Для backend сначала проверить DTO, сервис, контроллер и общий API type на frontend.

## Обязательные Проверки Перед Финалом

```bash
npm run format
npm run check
```

Для задач с UI или доступностью дополнительно, если среда позволяет:

```bash
npm run a11y
npm run ui:screenshot:docker
```

Если локальный Playwright/Chromium недоступен в среде, агент должен сначала попробовать Docker-вариант:

```bash
docker compose run --rm playwright npm run a11y
docker compose run --rm playwright npm run ui:screenshot
```

Если и он недоступен, агент должен явно написать, что автоматическая/визуальная проверка не выполнена и почему.

## Частые Ошибки AI-Агента В Этом Проекте

- Добавлять русский текст напрямую в шаблон вместо i18n-ключа.
- Дублировать модальные shell-стили вместо `ccs-modal-shell`.
- Создавать локальный tooltip вместо `ccsTooltip`.
- Добавлять hardcoded цвета/размеры в компонентный SCSS вместо design tokens.
- Использовать `app-` selector вместо `ccs-`.
- Оставлять бизнес-логику внутри `subscribe`.
- Добавлять новый тип рядом с компонентом, хотя он нужен нескольким компонентам или сервисам.
- Показывать одно и то же состояние в нескольких блоках страницы.
- Добавлять визуальный акцент к каждому блоку вместо одного главного действия.
- Менять UI без проверки `ACCESSIBILITY.md`.

## Как Формулировать Задачи Для AI

Хороший формат:

```text
Задача: улучшить UX /book.
Цель пользователя: быстро выбрать участников, дату, длительность и увидеть подходящие слоты.
Ограничения: не менять API, не добавлять новые библиотеки, сохранить i18n и accessibility.
Перед изменениями: прочитай AGENTS.md, AI_ONBOARDING.md, COMPONENTS.md, ACCESSIBILITY.md.
После изменений: npm run format, npm run check.
```

Для багов:

```text
Симптом: ...
Ожидаемое поведение: ...
Фактическое поведение: ...
Где смотреть: URL /..., файл/компонент если известен.
Проверь причину, исправь, обнови документацию если меняется правило проекта.
```

Для исследований:

```text
Проведи исследование, но сначала проверь текущий стек и ADR.
Если информация может быть устаревшей, используй актуальные официальные источники.
В конце дай решение, почему оно подходит этому проекту, и внеси изменения если они очевидны.
```

## Когда Обновлять Документацию

Документацию нужно обновлять, если меняется:

- пользовательский сценарий;
- API-контракт;
- схема БД;
- архитектурное решение;
- shared UI-компонент или правило его использования;
- i18n, accessibility или design-token правило;
- локальный запуск, Docker, Playwright или переменные окружения.
