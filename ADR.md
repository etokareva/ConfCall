# ADR — архитектурные решения ConfCall Scheduler

Для AI-агентов: перед изменением архитектуры, API-контрактов, схемы БД, навигации, shared UI, i18n или интеграций прочитать [AI_ONBOARDING.md](./AI_ONBOARDING.md) и relevant ADR ниже. Если принято новое устойчивое решение, добавить новый ADR в конец файла с новым номером.

## ADR-001. Angular SPA для frontend

Статус: принято

Контекст: приложению нужны несколько интерактивных экранов: доступность, поиск времени, встречи, настройки, публичное бронирование.

Решение: frontend реализован как Angular 19 SPA со standalone components, Router и lazy-loaded страницами.

Последствия:

- Страницы можно развивать независимо по feature-папкам.
- Router управляет основными сценариями.
- Для статического деплоя нужен SPA fallback через `.html` файлы.

## ADR-002. Standalone components и feature-based структура

Статус: принято

Контекст: проект небольшой, но уже содержит несколько предметных областей.

Решение: страницы и компоненты разложены по `frontend/src/app/features/*`, общие элементы лежат в `shared`, сервисы и модели API — в `core`.

Последствия:

- Проще находить код по пользовательскому сценарию.
- Shared-компоненты не зависят от конкретных страниц.
- Новые разделы стоит добавлять как отдельные feature-директории.

## ADR-003. RxJS в сервисах, signals в компонентах

Статус: принято

Контекст: Angular-приложение получает данные через HTTP и должно переиспользовать состояние между navbar, dashboard и страницами.

Решение: сервисы публикуют `Observable` и хранят кэш через `BehaviorSubject`/`ReplaySubject`; локальное UI-состояние страниц хранится в `signal()`.

Последствия:

- `async` pipe удобно использовать для серверных данных.
- Signals упрощают локальные формы, loading/error states и модалки.
- Нужно аккуратно синхронизировать POST-ответы с локальными subject, чтобы не делать лишние GET.

## ADR-004. NestJS modular backend

Статус: принято

Контекст: backend содержит разные предметные области: auth, availability, meeting, booking, video.

Решение: API построено на NestJS modules/controllers/services.

Последствия:

- Бизнес-логика остаётся в сервисах.
- Controllers отвечают за HTTP-контракт.
- Интеграции вроде Telemost скрыты за отдельным `VideoService`.

## ADR-005. Drizzle ORM и schema-first описание БД

Статус: принято

Контекст: нужна типизированная работа с MySQL без тяжёлого ORM-слоя.

Решение: схема описана в `db/schema.ts`, синхронизация выполняется через `drizzle-kit push`.

Последствия:

- Типы таблиц доступны в TypeScript.
- Локальная схема быстро обновляется через `npm run db:push`.
- Для production-миграций в будущем лучше перейти на явные SQL migration files и review миграций.

## ADR-006. MySQL как основная база данных

Статус: принято

Контекст: приложению нужны пользователи, слоты доступности, встречи, участники, booking-ссылки, группы и приглашения в группы.

Решение: используется MySQL 8.4. Для локальной разработки база поднимается через Docker Compose.

Последствия:

- Окружение воспроизводимо локально.
- `DATABASE_URL` обязателен для старта backend.
- SQLite не используется, потому что целевая модель и Drizzle-конфигурация ориентированы на MySQL.

## ADR-007. Регистрация и доступ только по приглашениям

Статус: принято

Контекст: пользователь не хочет открытую регистрацию “для всех”.

Решение: новые пользователи получают ссылку приглашения на email. Владелец группы подтверждает список email, после чего приложение отправляет письма с ссылкой на регистрацию или вход.

Последствия:

- Контроль доступа к приложению остаётся у администратора.
- Таблица `group_invitations` хранит email, токен, статус и дату принятия приглашения.
- Login page совмещает dev-вход администратора и вход по email/password, а регистрация происходит через ссылку приглашения.

## ADR-008. Dev-auth вместо полноценного OAuth на этапе локальной разработки

Статус: принято

Контекст: для разработки нужен быстрый вход без внешнего OAuth-провайдера.

Решение: `POST /api/auth/dev-login` создаёт или возвращает администратора `Dev User`, frontend хранит token `dev-token:<userId>` в `localStorage`.

Последствия:

- Локальная разработка не блокируется внешними провайдерами.
- Это не production-auth и должно быть заменено или ограничено перед production backend-деплоем.
- Backend endpoints получают пользователя из `Authorization: Bearer <token>`.

## ADR-009. Яндекс Телемост как текущая видеоплатформа

Статус: принято

Контекст: при создании встречи должна появляться реальная ссылка на видеозвонок, а mock-ссылки не подходят.

Решение: `VideoService` вызывает `POST https://cloud-api.yandex.net/v1/telemost-api/conferences` с `Authorization: OAuth <TELEMOST_OAUTH_TOKEN>`.

Последствия:

- Нужен OAuth access token с scope `telemost-api:conferences.create`.
- Токен должен быть выпущен пользователем организации Яндекс 360 для бизнеса.
- Ошибки 401/403 переводятся в понятные сообщения о настройке токена.
- Встреча в ConfCall хранит `videoUrl` и `videoMeetingId`.

## ADR-010. Поиск пересечения доступности на backend

Статус: принято

Контекст: логика пересечения зависит от доступности участников и уже созданных встреч.

Решение: алгоритм находится в `AvailabilityService`, а frontend только вызывает API и показывает результат.

Последствия:

- Единый источник бизнес-логики.
- Frontend остаётся тонким.
- Ошибки вроде “нет доступности” или “нет пересечения” должны приходить как понятные сообщения.

## ADR-011. POST endpoints возвращают обновлённые данные

Статус: принято

Контекст: после сохранения доступности, создания встречи или изменения booking-ссылки frontend не должен делать лишний GET.

Решение: mutation endpoints возвращают созданный/обновлённый ресурс или достаточный payload для обновления локального состояния.

Последствия:

- Меньше сетевых запросов.
- Сервисы frontend обновляют `BehaviorSubject` локально.
- Контракты POST-ответов становятся частью UX и должны поддерживаться аккуратно.

## ADR-012. Новые встречи хранятся через `meeting_participants.seenAt`

Статус: принято

Контекст: пользователь должен видеть бейдж новых встреч, если его добавили участником.

Решение: у участника встречи есть `seenAt`. Если текущий пользователь связан с participant и `seenAt` пустой, встреча считается новой.

Последствия:

- Бейдж новых встреч работает между сессиями.
- При открытии страницы встреч вызывается `POST /api/meetings/mark-seen`.
- Для корректной работы участник должен быть связан с пользователем по email/userId.

## ADR-013. Публичные booking-ссылки групп

Статус: принято

Контекст: группе нужна возможность дать внешнему гостю ссылку, чтобы гость без аккаунта увидел общую доступность участников группы и сам выбрал свободное время.

Решение: `booking_links` содержит `slug`, `groupId`, настройки длительности и активность. Ссылки создаются и управляются в `/book` после выбора группы и конкретных участников. Таблица `booking_link_participants` хранит выбранных участников ссылки, поэтому публичное бронирование может быть создано не для всей группы, а для нужной части участников. Публичная страница `/book/:slug` использует slug для загрузки данных и показывает аватар группы, участников этой ссылки, подсвеченные даты со свободным временем, доступные слоты и форму гостя.

Последствия:

- Гость не обязан быть зарегистрирован.
- Встреча создаётся для выбранных участников публичной ссылки и гостя, который оставил контактные данные.
- После успешного бронирования гостю отправляется письмо со ссылкой на видеозвонок и публичной ссылкой отмены `/book/cancel/:token`.
- Публичная отмена гостем отменяет встречу целиком, потому что гость является инициатором внешнего созвона с группой.
- `meeting_participants.cancelToken` хранит токен гостя, а `meetings.cancelledByName/cancelledByEmail/cancelledAt` фиксируют автора отмены для отображения всем участникам.
- Создание ссылки находится в `/book` после выбора группы, участников и доступного времени; управление созданными ссылками находится в `/booking-links`. URL ссылки должен быть кликабельным, а действия `open/copy/send/toggle/delete` отображаются компактными иконками с tooltip и aria-label.
- Любой участник группы с доступом к ссылке может управлять group-scoped booking links в рамках разрешений backend.
- Для старых ссылок без `booking_link_participants` backend использует fallback на всех участников группы.
- Неактивная ссылка возвращает `410 Gone` и `messageKey: "booking.link_inactive"`, frontend показывает локализованное временное состояние без формы бронирования и кнопку повторной проверки. Если ссылку снова включили, тот же `slug` снова открывается и доступен для бронирования.
- `GET /api/booking/public/:slug` возвращает плоский `group: { id, name, avatar, members } | null`; frontend не должен ожидать вложенный `{ group, members }`.
- Владелец или участники встречи получают новую встречу как непросмотренную по общему механизму `meeting_participants.seenAt`.

## ADR-014. Docker Compose для локальной инфраструктуры

Статус: принято

Контекст: MySQL и Playwright должны запускаться воспроизводимо.

Решение: `docker-compose.yml` содержит `mysql` и `playwright` services.

Последствия:

- MySQL поднимается командой `npm run db:up`.
- Playwright запускается в контейнере с готовыми браузерами.
- На некоторых машинах Docker требует `sudo` или членство пользователя в группе `docker`.

## ADR-015. Playwright screenshots через Docker

Статус: принято

Контекст: нужно смотреть интерфейс и делать скриншоты без ручной установки браузеров.

Решение: используется `@playwright/test` и официальный Docker image `mcr.microsoft.com/playwright:v1.60.0-noble`.

Последствия:

- Команда `npm run ui:screenshot:docker` делает PNG основных страниц.
- Скриншоты складываются в `artifacts/screenshots/`.
- Generated screenshots и reports не коммитятся.

## ADR-016. Единый ModalService и ModalShell вместо native alert/confirm

Статус: принято

Контекст: native browser dialogs выглядят чужеродно и плохо контролируются. Кроме простых подтверждений в приложении есть содержательные диалоги: бронирование встречи, детали встречи, настройка слота и расписание дня. Локальные shell/backdrop/header стили быстро расходятся и создают ошибки с фокусом, z-index и расположением кнопки закрытия.

Решение: добавлены `ModalService`, `ModalComponent` и общий `ccs-modal-shell`.

- `ModalService` использовать для простых `info`, `error`, `confirm`, `choose`.
- `ccs-modal-shell` использовать как каркас всех содержательных модальных окон: заголовок, eyebrow, subtitle, кнопка закрытия, header actions, body и footer.
- Содержимое диалога остаётся в feature-компоненте, но card/header/footer/close берутся из общего shell.

Последствия:

- Подтверждения, ошибки и информационные сообщения выглядят единообразно.
- Страницы вызывают `modal.info`, `modal.error`, `modal.confirm`.
- Новые модальные окна не должны создавать собственные `.dialog-close`, `.dialog-header`, shell-card и shadow-стили.
- Focus trap, initial focus, backdrop и overlay-настройки должны оставаться на CDK Dialog или общем модальном слое.

## ADR-017. Очистка transient UI при навигации

Статус: принято

Контекст: тосты и модальные окна относятся к текущему пользовательскому сценарию. Если пользователь переходит на другой маршрут, старые уведомления и диалоги могут быть уже неактуальны, мешать фокусу, screen reader announcements и создавать ощущение, что действие относится к новой странице.

Решение: корневой `AppComponent` слушает `NavigationStart` и очищает transient UI:

- закрывает все CDK Dialog окна через `Dialog.closeAll()`;
- очищает активные и ожидающие toast-уведомления через `ToastService.clear()`.

Последствия:

- Новые модальные окна должны открываться через CDK Dialog или `ModalService`, чтобы автоматически закрываться при смене маршрута.
- Toast-уведомления должны создаваться через `ToastService`, чтобы очередь, таймеры и overlay очищались централизованно.
- Долгоживущие UI-элементы, которые должны переживать навигацию, нельзя строить поверх этого transient overlay-паттерна без отдельного архитектурного решения.

## ADR-018. Статический деплой SPA с fallback-файлами

Статус: принято

Контекст: текущий хостинг отдаёт статические файлы и не поддерживает полноценную server-side rewrite настройку.

Решение: для маршрутов создаются `.html` fallback-файлы, которые являются копией `index.html`.

Последствия:

- Прямое открытие `/book` может работать как `/book.html`.
- Нужно помнить обновлять fallback-файлы при добавлении публичных маршрутов.
- Для production backend deployment это решение может быть пересмотрено в пользу reverse proxy.

## ADR-019. Единые SCSS design tokens

Статус: принято

Контекст: стили приложения начали расходиться: цвета, радиусы, размеры контролов и отступы повторялись в разных component SCSS.

Решение: базовые визуальные значения вынесены в `frontend/src/styles/_tokens.scss` как CSS custom properties. Глобальный `styles.scss` подключает tokens через `@use`.

Последствия:

- Компоненты переиспользуют `var(--color-*)`, `var(--space-*)`, `var(--radius-*)`, `var(--control-height)` и другие tokens.
- Новые hardcoded `#hex`/`rgb()` цвета в component SCSS не добавляются без причины.
- Смена темы или ключевых размеров теперь делается централизованно.

## ADR-020. flatpickr для выбора даты бронирования

Статус: принято

Контекст: кастомный календарь потребовал бы много кода для позиционирования, ввода с клавиатуры, локализации и edge cases.

Решение: используется общий `ccs-date-input` поверх `flatpickr` с runtime-локалью, ручным вводом даты в локальном формате, валидацией даты и стилизацией через глобальные tokens. Внутреннее значение control и API-контракт для дат — date-only ISO `YYYY-MM-DD`.

Последствия:

- Календарь открывается относительно поля даты.
- Пользователь может выбрать дату мышью или ввести её вручную.
- Стили календаря нужно поддерживать в общем визуальном языке приложения.
- Для сценариев с известной доступностью `ccs-date-input.availableDates` подсвечивает даты через общий класс `ccs-flatpickr-day--available`. Этот механизм используется на публичной странице бронирования и на `/book`: для одного участника подсвечиваются его свободные дни, для нескольких — дни с общими пересечениями.
- Feature-компоненты не должны дублировать vendor-overrides flatpickr; общие стили лежат в `frontend/src/styles/vendors/_flatpickr.scss`.

## ADR-021. Аватар группы в публичном бронировании

Статус: принято

Контекст: публичная страница бронирования должна сразу показывать гостю, с какой группой он бронирует встречу. Одних имени группы и списка участников недостаточно для узнаваемости, особенно если ссылок несколько.

Решение: у таблицы `groups` есть nullable поле `avatar`. Владелец группы может сохранить URL изображения группы в `/groups`. Публичный endpoint booking возвращает этот URL в `group.avatar`, а frontend показывает изображение в header публичной страницы. Если URL пустой или картинка недоступна, используется смысловая иконка `users`.

Последствия:

- `POST /api/groups/:id` обновляет `name` и `avatar`; доступно только владельцу группы.
- Все UI-места с классом `avatar` должны сначала пытаться показать реальное изображение, если URL есть и загружается.
- Фолбэк на инициалы не используется для группы, потому что для группы лучше подходит иконка `users`.
