# Улучшение алгоритма поиска пересечений

## Назначение документа

Документ фиксирует, как работал поиск общего времени до оптимизации, какие изменения были внесены в схему базы данных и как это повлияло на производительность алгоритма.

Связанные файлы:

- `server/src/availability/availability.service.ts` — бизнес-логика поиска пересечений.
- `db/schema.ts` — схема Drizzle ORM.
- `db/migrations/0005_add_intersection_indexes.sql` — SQL-миграция с индексами.

## Как работал алгоритм до оптимизации

Поиск пересечений выполнялся в несколько этапов:

1. Backend получал список выбранных пользователей.
2. Для конкретной даты определялся день недели.
3. Из `availability_slots` загружались weekly-слоты выбранных пользователей.
4. Из `availability_calendar_events` загружались календарные слоты выбранных пользователей.
5. Из `meetings` загружались встречи, пересекающиеся с выбранной датой или диапазоном.
6. Из `meeting_participants` загружались участники найденных встреч.
7. В памяти приложения строились свободные интервалы каждого пользователя.
8. Интервалы последовательно пересекались между собой.
9. Из общего пересечения вычитались занятые интервалы встреч.
10. Оставшиеся интервалы возвращались как `availableRanges`; если задана длительность, они нарезались на `availableSlots`.

Основная проблема была не в самом алгоритме пересечения интервалов, а в доступе к данным. До добавления индексов MySQL мог читать таблицы целиком и только потом фильтровать строки по пользователям, дате, статусу и встречам.

Упрощённо:

```text
availability_slots       -> scan по всей таблице
availability_events      -> scan по всей таблице
meetings                 -> scan по всей таблице
meeting_participants     -> scan по всей таблице
group_members            -> scan по всей таблице при проверках доступа
```

На маленькой базе это почти незаметно, но при росте количества пользователей, групп, слотов и встреч такой подход приводит к линейному росту времени ответа.

Вторая проблема была в пересечении интервалов в памяти. Раньше `intersectRanges(left, right)` сравнивал каждый интервал из левого списка с каждым интервалом из правого списка:

```text
for leftRange in left:
  for rightRange in right:
    check overlap
```

Для двух списков длиной `L` и `R` это давало сложность:

```text
O(L * R)
```

Если у пользователей много свободных слотов в течение дня, количество сравнений быстро растёт.

## Что было сделано

В `db/schema.ts` добавлены составные индексы для таблиц, которые участвуют в поиске пересечений и проверках доступа. После этого в `AvailabilityService` добавлен SQL-фильтр по пересечению дат для `availability_calendar_events`, чтобы не загружать заведомо нерелевантные календарные правила.

### Индексы доступности

```text
availability_slots_user_day_active_idx
  availability_slots(userId, dayOfWeek, isActive)
```

Этот индекс ускоряет поиск weekly-слотов конкретных пользователей на конкретный день недели.

```text
availability_calendar_events_user_dates_active_idx
  availability_calendar_events(userId, startDate, endDate, isActive)
```

Этот индекс нужен для календарных слотов: разовых и повторяющихся правил доступности. Он позволяет быстрее отбирать события выбранных пользователей и в дальнейшем эффективнее фильтровать их по диапазону дат.

### SQL-фильтр календарных событий по диапазону дат

До доработки backend загружал все активные календарные события выбранных пользователей, а затем проверял каждое событие в TypeScript:

```text
calendarEvents -> загрузить все активные -> calendarEventApplies(...) в JS
```

Теперь события сначала отбираются на уровне SQL:

```sql
userId IN (...)
AND isActive = true
AND startDate <= :endDateQuery
AND endDate >= :startDateQuery
```

Это стандартная проверка пересечения двух диапазонов:

```text
событие: [startDate, endDate]
запрос:  [startDateQuery, endDateQuery]

диапазоны пересекаются, если:
startDate <= endDateQuery
AND
endDate >= startDateQuery
```

Если хотя бы одно из условий не выполняется, событие точно не может повлиять на результат поиска:

- `startDate > endDateQuery` — событие начинается после конца запрошенного периода;
- `endDate < startDateQuery` — событие закончилось до начала запрошенного периода.

После SQL-отбора TypeScript всё ещё проверяет повторяемость через `calendarEventApplies(...)`. Это важно, потому что SQL-фильтр отвечает только на вопрос “может ли событие попасть в период?”, а JS-логика отвечает на вопрос “срабатывает ли повторяющееся правило именно в этот день?”.

Для range-поиска календарные события теперь загружаются один раз на весь диапазон и переиспользуются при расчёте каждого дня. Это убирает повторные одинаковые запросы к `availability_calendar_events` для каждой даты диапазона.

### Batch-загрузка данных для range-поиска

До доработки range-поиск строился как набор независимых дневных расчётов:

```text
для каждого дня диапазона:
  загрузить weekly-слоты
  загрузить calendar events
  загрузить встречи
  загрузить участников встреч
  посчитать пересечение
```

Даже при диапазоне в 31 день это могло создавать десятки однотипных SQL-запросов.

Теперь данные, которые нужны для всего диапазона, загружаются заранее:

```text
один раз на диапазон:
  weekly-слоты выбранных пользователей для нужных дней недели
  calendar events, пересекающиеся с диапазоном
  busy meetings выбранных пользователей, пересекающиеся с диапазоном
  participants для найденных busy meetings

для каждого дня:
  отфильтровать уже загруженные данные
  посчитать пересечение
```

Это уменьшает количество SQL-запросов для range-поиска. Вместо повторения одинаковых запросов на каждый день backend делает ограниченный набор batch-запросов и дальше работает с данными в памяти.

### Предварительная нормализация busy intervals

До доработки backend сначала загружал все `scheduled` встречи в диапазоне, а затем в памяти проверял, относятся ли они к выбранным пользователям:

```text
meetings in range -> filter by organizer or participant in JS
```

Теперь в расчёт попадают только встречи, которые могут занять время выбранных пользователей:

1. Встречи, где `organizerId` входит в `userIds`.
2. Встречи, где выбранные пользователи есть в `meeting_participants` со статусом `pending` или `confirmed`.

Для второго случая используется join `meeting_participants -> meetings`, поэтому база сразу применяет фильтры по участникам, статусу участия, статусу встречи и диапазону времени.

После этого обе выборки объединяются по `meeting.id`, чтобы одна и та же встреча не попала в busy intervals дважды, если пользователь одновременно организатор и участник.

### Пересечение интервалов через sort + merge + two-pointer

Функция `intersectRanges(...)` заменена на более эффективный алгоритм:

1. Список интервалов каждого пользователя сортируется по началу и окончанию.
2. Пересекающиеся или соприкасающиеся интервалы одного пользователя склеиваются через `mergeRanges(...)`.
3. Два отсортированных списка пересекаются двумя указателями.
4. На каждом шаге сравниваются только текущие интервалы, после чего двигается указатель списка, чей интервал заканчивается раньше.
5. Результат возвращается без повторной нормализации, потому что входы уже отсортированы и склеены, а two-pointer сохраняет порядок результата.

Упрощённо:

```text
left  = merge(sort(left))
right = merge(sort(right))

i = 0
j = 0

while i < left.length and j < right.length:
  overlap = intersection(left[i], right[j])
  if overlap exists:
    result.push(overlap)

  if left[i].end < right[j].end:
    i += 1
  else:
    j += 1
```

Сложность стала:

```text
sort:        O(L log L + R log R)
merge:       O(L + R)
two-pointer: O(L + R)
```

После нормализации повторные пересечения между несколькими пользователями работают на более коротких списках, потому что перекрывающиеся интервалы одного пользователя уже склеены.

Важно: `intersectRanges(...)` больше не вызывает `mergeRanges(...)` внутри себя. Инвариант теперь такой: все входные списки должны быть нормализованы до начала цепочки пересечений. Это убирает повторную сортировку и склейку уже нормализованных списков.

### Ограничение диапазона поиска

Для range-поиска добавлено ограничение: один запрос может покрывать не больше 31 календарного дня.

Причины:

1. Пользовательский сценарий обычно связан с поиском времени в ближайшие дни или в рамках месяца.
2. Слишком широкий диапазон может случайно создать десятки или сотни дневных расчётов.
3. Backend всё ещё считает каждый день отдельно, но данные для расчёта загружаются batch-запросами на весь диапазон.
4. Ограничение даёт предсказуемую верхнюю границу нагрузки на API.

Проверка добавлена в двух местах:

- backend возвращает `availability.intersection.range_too_long`, если API получает слишком широкий диапазон;
- frontend показывает локализованную ошибку формы и не отправляет запрос, если диапазон больше 31 дня.

### Короткоживущий кэш одинаковых read-запросов

Для `GET /availability/intersection` добавлен in-memory cache с TTL 5 секунд.

Кэш применяется только после проверок доступа к группе и участникам. Ключ строится из нормализованных параметров запроса:

- `userIds`, отсортированные по возрастанию;
- `date`;
- `startDate`;
- `endDate`;
- `durationMinutes`;
- `groupId`.

Это помогает, когда frontend или пользовательский сценарий создаёт несколько одинаковых read-запросов подряд: например, при реактивном обновлении формы, повторном открытии шага или быстром переключении UI-состояний без изменения параметров поиска.

Кэш очищается при изменении availability внутри `AvailabilityService`:

- `setSlots(...)`;
- `setEvents(...)`.

Для изменений в других сервисах добавлена cross-service инвалидация через `CacheInvalidationService`. Он не связывает сервисы напрямую: сервисы, которые меняют данные, публикуют событие `invalidateIntersections()`, а `AvailabilityService` подписывается на него и очищает свой in-memory cache.

Для нескольких backend-инстансов добавлена общая версия кэша в MySQL:

```text
cache_versions(name, version, updatedAt)
```

При каждой инвалидации версия `intersection` увеличивается в БД. Перед чтением локального in-memory cache `AvailabilityService` получает текущую версию из MySQL и сравнивает её с версией cache entry. Если версии отличаются, локальное значение удаляется и пересчитывается. Поэтому разные backend-инстансы могут иметь собственные локальные кэши, но валидность этих кэшей определяется общей версией в БД.

Кэш сбрасывается после успешных изменений:

- `meetings`;
- `meeting_participants`;
- `group_members`.

TTL оставлен коротким, потому что результат поиска пересечений зависит от часто меняющихся сущностей:

- слоты доступности участников;
- календарные правила доступности;
- созданные, отменённые или перенесённые встречи;
- состав группы и выбранные участники;
- длительность встречи.

Короткий TTL остаётся полезным как дополнительная защита: если в будущем появится новый write-сценарий и в нём забудут вызвать инвалидацию, устаревшее значение проживёт недолго.

### Индексы групп

```text
group_members_group_user_idx
  group_members(groupId, userId)

group_members_user_group_idx
  group_members(userId, groupId)
```

Эти индексы ускоряют проверку членства пользователя в группе и выбор участников группы. Это важно, потому что бронирование и поиск пересечений ограничены выбранной группой.

### Индексы встреч

```text
meetings_status_time_idx
  meetings(status, startTime, endTime)

meetings_organizer_status_time_idx
  meetings(organizerId, status, startTime, endTime)
```

Эти индексы ускоряют поиск запланированных встреч, которые пересекаются с выбранной датой или диапазоном. Такие встречи нужно вычитать из свободного времени.

### Индексы участников встреч

```text
meeting_participants_meeting_user_status_idx
  meeting_participants(meetingId, userId, status)

meeting_participants_user_status_meeting_idx
  meeting_participants(userId, status, meetingId)
```

Эти индексы ускоряют поиск встреч, где выбранные пользователи являются участниками, и позволяют быстрее отбрасывать отменённые или отклонённые участия.

## Как изменилась сложность доступа к данным

До индексов база могла выполнять полное чтение таблиц:

```text
O(все строки таблицы)
```

После индексов база может искать только релевантные строки:

```text
O(log N + найденные строки)
```

Это особенно важно для таблиц, которые со временем будут расти быстрее всего:

- `meetings`
- `meeting_participants`
- `availability_calendar_events`
- `group_members`

## Результаты проверки на локальной базе

На локальной базе данных объём сейчас очень маленький:

```text
availability_slots: 3 строки
availability_calendar_events: 5 строк
meetings: 5 строк
meeting_participants: 10 строк
group_members: 4 строки
```

Из-за этого абсолютные замеры в миллисекундах не являются показательными: на 5-10 строках MySQL иногда быстрее делает полный scan таблицы, чем обращается к индексу.

Тем не менее `EXPLAIN ANALYZE` показал, что для ключевого запроса weekly-доступности индекс уже используется:

```text
availability_slots по userId + dayOfWeek + isActive

с индексом:    ~0.48 ms, index range scan
без индекса:  ~0.96 ms, table scan
```

На текущем объёме это примерно в 2 раза быстрее для этого запроса.

Для остальных таблиц эффект на маленькой базе пока не раскрывается полностью:

```text
availability_calendar_events: таблица слишком маленькая, full scan дешевле
meetings: 5 строк, разницы почти нет
meeting_participants: 10 строк, оптимизатор может выбирать full scan
```

Это нормальное поведение MySQL: оптимизатор выбирает самый дешёвый план с учётом статистики таблиц. Индексы начинают давать заметный выигрыш, когда данных становится достаточно много.

## Что улучшилось для продукта

1. Поиск пересечений стал готов к росту данных.
2. Запрос weekly-доступности уже перешёл с table scan на index range scan.
3. Проверки групп и участников получили индексы под реальные сценарии приложения.
4. Поиск занятых встреч получил индексы по статусу и времени.
5. Схема базы стала явно отражать основные read-сценарии приложения.
6. Календарные события доступности теперь отбираются по пересечению дат в SQL, а не полностью фильтруются в памяти приложения.
7. Для поиска по диапазону дат weekly-слоты, календарные события, встречи и участники встреч загружаются batch-запросами на весь диапазон.
8. Пересечение интервалов в памяти больше не использует вложенный цикл `left × right`; вместо него используется `sort + merge + two-pointer`.
9. Range-запрос ограничен 31 днём, чтобы API не выполнял слишком широкий расчёт одним запросом.
10. Busy intervals нормализуются заранее: backend получает только встречи, где выбранные пользователи являются организаторами или участниками.
11. Уже нормализованные списки интервалов не сортируются и не склеиваются повторно внутри цепочки пересечений.
12. Для одинаковых read-запросов `/availability/intersection` добавлен короткоживущий in-memory cache с TTL 5 секунд.
13. Кэш пересечений инвалидируется между сервисами при изменениях `meetings`, `meeting_participants` и `group_members`.
14. Для нескольких backend-инстансов добавлена shared DB-version invalidation через таблицу `cache_versions`.

## Ограничения текущей оптимизации

Главное оставшееся ограничение — range-поиск всё ещё считает каждый день отдельно. Данные для диапазона уже загружаются batch-запросами, но сам расчёт пересечения выполняется по каждому дню, потому что weekly-правила, разовые/повторяющиеся календарные правила и занятые встречи нужно привести к конкретной календарной дате.

Это корректно функционально и больше не создаёт лишние обращения к базе на каждый день, но при большом диапазоне всё ещё увеличивает количество вычислений в памяти.

Кэширование остаётся локальным по хранению значений: сами ответы лежат в памяти конкретного backend-процесса. Общей является версия валидности в MySQL. Это проще, чем Redis/pub-sub, и достаточно для короткоживущего read-cache, но каждый инстанс всё равно пересчитывает значение самостоятельно после инвалидированной версии.

## Следующие возможные улучшения

1. Добавить Redis или другой общий cache backend, если понадобится хранить сами ответы кэша между несколькими backend-инстансами.
2. Добавить benchmark fixture с большим количеством тестовых данных: например 100 пользователей, 1000 слотов, 10000 встреч.
3. Добавить автоматический performance-smoke test для `/api/availability/intersection`.
4. Проверить необходимость дополнительных covering indexes после появления реальных объёмов данных.

## Вывод

Добавленные индексы, SQL-фильтр по пересечению дат, batch-загрузка range-данных, предварительный отбор busy meetings, короткоживущий read-cache, shared DB-version invalidation и замена `intersectRanges` на `sort + merge + two-pointer` не меняют пользовательское поведение. Они улучшают два слоя алгоритма: доступ к данным и вычисление пересечений в памяти.

На текущей маленькой базе видимый выигрыш есть только на запросе weekly-доступности: примерно с `0.96 ms` до `0.48 ms`. Главная польза изменения проявится при росте данных: вместо полного чтения таблиц MySQL сможет использовать индексные выборки по пользователям, датам, статусам, группам и встречам. Для календарных событий особенно важно, что теперь в приложение не попадают правила, которые точно не пересекаются с выбранной датой или диапазоном.
